## DevOps Review — 2026-05-18

### Summary

The infrastructure shows a solid foundation: remote state is properly hosted in Azure Blob Storage, managed identity is used throughout, OIDC authentication is wired into the pipeline, and sensitive variables are correctly marked. The most critical gap is the PostgreSQL Flexible Server having no explicit public network access restriction and no VNet integration — the server is exposed to the internet with no Terraform-enforced firewall rules. Close behind that are three High findings that compound risk: all GitHub Actions references use floating version tags (supply chain attack surface), the Log Analytics workspace is provisioned but receives zero logs (no diagnostic settings exist on any resource), and the Container App has no `ingress` block, meaning the API is not publicly reachable. Quick wins are dependency caching in the CI pipeline and adding `prevent_destroy` lifecycle blocks to critical infrastructure resources.

### Findings

| # | Severity | Category | Resource | Finding | Recommendation |
|---|----------|----------|----------|---------|----------------|
| 1 | **Critical** | Security | `azurerm_postgresql_flexible_server.timezonemanagementserver` | No `public_network_access_enabled = false`, no VNet/subnet delegation, no `azurerm_postgresql_flexible_server_firewall_rule` defined. The server is internet-reachable and relies solely on Azure's default deny-all behavior — not enforced or auditable via Terraform. | Add `public_network_access_enabled = false` to the server resource, or configure `delegated_subnet_id` + `private_dns_zone_id` for VNet-private access. Enforce access rules in Terraform rather than relying on portal defaults. |
| 2 | **High** | Security | `.github/workflows/ci-cd-pipeline.yml` — all actions | Every action reference uses a floating major-version tag (`@v6`, `@v4`, `@v5`, `@v7`, `@v2`, `@v3`). Floating tags can be silently overwritten, enabling a supply chain attack to inject malicious code into the pipeline. | Pin each action to a full commit SHA (e.g., `actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v6.0.2`). Maintain a pinning table and use Dependabot or Renovate to keep SHAs current. |
| 3 | **High** | Security | `azurerm_key_vault_access_policy.terraform_sp` | The Terraform service principal is granted `Purge` on Key Vault secrets. `Purge` is an irreversible operation that permanently removes a soft-deleted secret, bypassing the `soft_delete_retention_days = 7` safeguard. | Remove `Purge` from the access policy. Purge capability is only needed for `terraform destroy` cleanup and should be exercised manually or via a dedicated break-glass principal. |
| 4 | **High** | Azure Compliance | `azurerm_container_app.container_app` | No `ingress` block is defined. Without ingress, the Container App accepts no external traffic — the API is not publicly reachable despite being deployed. The `deploy` job will update the image but clients cannot connect. | Add an `ingress` block with `external_enabled = true`, `target_port = <app_port>`, and `transport = "auto"` (or `"http"`) to expose the API externally. |
| 5 | **High** | Observability | All resources | `azurerm_log_analytics_workspace.log_analytic` is provisioned but no `azurerm_monitor_diagnostic_setting` is defined for any resource (Container App, Key Vault, App Configuration, PostgreSQL). The workspace receives no data. | Add `azurerm_monitor_diagnostic_setting` for at minimum: the Container App environment, Key Vault, and PostgreSQL server, forwarding `allLogs` and `AllMetrics` categories to the Log Analytics workspace ID. |
| 6 | **High** | CI/CD | `build`, `tests`, `pushImageToDockerHub` jobs | No `actions/cache` step for NuGet packages. Each job independently runs `dotnet restore`, re-downloading the entire dependency graph from NuGet every run. With three jobs all restoring separately, this wastes significant runner minutes on every push to `main`. | Add a cache step before `dotnet restore` in each job: `actions/cache@v4` keyed on `${{ hashFiles('**/*.csproj') }}` with `~/.nuget/packages` as the path. |
| 7 | **High** | Terraform BP | `required_providers.azurerm` (both `main.tf` and `bootstrap/required_main.tf`) | Provider is pinned to `=4.1.0`, which was released in September 2024 — approximately 8 months behind the current latest `4.x` release. This misses security patches and new resource arguments for azurerm resources used in this configuration. | Update to `~> 4.1` (allow patch upgrades within 4.1.x) or evaluate upgrading to the latest `4.x`. Run `terraform init -upgrade` after changing the constraint and review the changelog for breaking changes. |
| 8 | **Medium** | Security | `azurerm_key_vault.app_config_key_vault` | Uses the legacy access policy authorization model (`azurerm_key_vault_access_policy`) instead of Azure RBAC. Microsoft recommends RBAC for new Key Vaults as it offers better auditability, conditional access support, and deny assignments. | Set `enable_rbac_authorization = true` on the Key Vault and replace `azurerm_key_vault_access_policy` resources with `azurerm_role_assignment` using built-in roles (`Key Vault Secrets Officer` for the SP, `Key Vault Secrets User` for the Container App). |
| 9 | **Medium** | Terraform BP | `terraform.required_version = ">= 1.5.2"` | The floor constraint `>= 1.5.2` accepts any future Terraform version including a hypothetical 2.x, which may introduce breaking changes. The current pipeline installs 1.15.3 but nothing in the config prevents accidental use of an incompatible version. | Tighten to `>= 1.5.2, < 2.0.0` or `~> 1.5` to explicitly cap at the current major version boundary. |
| 10 | **Medium** | Terraform BP | `infrastructure/bootstrap/required_main.tf` | Bootstrap has no `backend` block — Terraform state for the shared Resource Group and storage account is local only. If the developer's machine is lost or the local state file is deleted, Terraform loses track of these critical resources and future `plan`/`apply` runs will try to recreate them. | Store bootstrap state in the same Azure Blob Storage container (using a separate `key`, e.g., `bootstrap.tfstate`). Note: this creates a chicken-and-egg dependency; the standard workaround is to create the storage account manually once, import it, then enable the backend. |
| 11 | **Medium** | Terraform BP | `azurerm_key_vault`, `azurerm_postgresql_flexible_server`, `azurerm_container_app` | No `lifecycle { prevent_destroy = true }` on Key Vault, PostgreSQL server, or Container App. Only the database resource has this protection. A misfire of `terraform destroy` or an accidental resource replacement could permanently delete these. | Add `lifecycle { prevent_destroy = true }` to `azurerm_key_vault.app_config_key_vault`, `azurerm_postgresql_flexible_server.timezonemanagementserver`, and `azurerm_container_app.container_app`. |
| 12 | **Medium** | Observability | (global) | No `azurerm_monitor_metric_alert` or `azurerm_monitor_scheduled_query_rules_alert` resources defined. There is no alerting on PostgreSQL CPU/connection saturation, Container App restart loops, or Key Vault access failures. | Define at minimum: a metric alert on PostgreSQL CPU > 80% and a Container App restart-count alert. These are low-cost but critical for production visibility. |
| 13 | **Medium** | CI/CD | `build` / `tests` jobs | The `build` job compiles the solution but uploads no artifact. The `tests` job checks out fresh, re-restores, and implicitly re-builds via `dotnet test`. The `build` job's compiled output is discarded, providing validation-only value with no downstream reuse. | Either publish the build output as an artifact with `dotnet publish` + `actions/upload-artifact` and consume it in `tests`, or collapse `build` and `tests` into a single job to eliminate redundant work. |
| 14 | **Low** | Cost/Tagging | `azurerm_log_analytics_workspace.log_analytic` | Missing `tags` block. All other resources apply `local.tags` (environment, project, owner, managed_by), but the workspace is untagged, which breaks cost allocation consistency. | Add `tags = local.tags` to the workspace resource. |
| 15 | **Low** | CI/CD | `tests` job | Test navigation uses two sequential `cd` commands (`cd tests; cd Timezone.Management.Application.Tests`) instead of a single invocation. Minor readability issue and fragile to directory renames. | Replace with `dotnet test tests/Timezone.Management.Application.Tests` from the repo root, which is consistent with the `CLAUDE.md` command documentation. |

### Positive Observations

- **Remote state backend**: The main configuration correctly uses `backend "azurerm"` with a dedicated blob container — no local state for team-shared infrastructure.
- **Sensitive variable handling**: `db_login` and `db_password` are both marked `sensitive = true`, and the constructed connection string is stored in Key Vault rather than passed directly to the Container App.
- **Variable quality**: All variables in both `variables.tf` files have `type`, `description`, and validation constraints where appropriate.
- **Managed identity throughout**: The Container App uses `type = "SystemAssigned"` identity — no service principal credentials stored in the application.
- **Least-privilege RBAC on Container App**: The app is granted only `App Configuration Data Reader` and Key Vault `Get`/`List` — exactly what it needs.
- **Key Vault hardening**: `purge_protection_enabled = true` and `soft_delete_retention_days = 7` are both set.
- **Bootstrap resource locks**: Both the Resource Group and storage account have `azurerm_management_lock` with `CanNotDelete` — preventing accidental deletion of the Terraform backend.
- **Storage account security**: `min_tls_version = "TLS1_2"`, `https_traffic_only_enabled = true`, and blob versioning with a 7-day retention policy.
- **Container storage is private**: `container_access_type = "private"` on the Terraform state container.
- **OIDC authentication**: Azure login uses OIDC federation (`client-id` + `tenant-id` + `subscription-id`) with no long-lived secrets. `id-token: write` permission is correctly set at the workflow level.
- **Environment gates**: Both `terraformApply` and `deploy` jobs declare `environment: production`, enabling branch protection rules and manual approval gates.
- **Artifact version consistency**: `upload-artifact@v7` and `download-artifact@v7` use matching major versions, avoiding protocol mismatches.
- **Terraform version is current**: The pipeline installs `1.15.3`, which is the latest stable Terraform release as of 2026-05-18 and satisfies the `>= 1.5.2` constraint.
- **Database protect_destroy**: `azurerm_postgresql_flexible_server_database` has `lifecycle { prevent_destroy = true }` — the most important resource for data safety is protected.
- **Consistent tagging**: All resources except the Log Analytics workspace apply `local.tags` with `environment`, `project`, `owner`, and `managed_by` keys.
- **Docker credentials**: DockerHub token is passed via `secrets.DOCKERHUB_TOKEN`; username via `vars.DOCKERHUB_USERNAME` — correct separation of sensitive vs. non-sensitive config.

### Next Steps

1. **[Critical]** Restrict PostgreSQL network access: add `public_network_access_enabled = false` and evaluate VNet integration for the server. This is the highest-risk change and should be done before any production traffic.
2. **[High]** Add `ingress` block to the Container App — without this the deployed API is unreachable, making every deployment effectively a no-op.
3. **[High]** Pin all GitHub Actions to commit SHAs. Use `gh api` or the GitHub UI to get each action's current SHA, then configure Dependabot (`dependabot.yml`) to keep them updated automatically.
4. **[High]** Add `azurerm_monitor_diagnostic_setting` resources for Container App environment, Key Vault, and PostgreSQL. The Log Analytics workspace is already provisioned — connecting resources to it is low-effort and immediately enables observability.
5. **[High]** Remove `Purge` from the Terraform SP Key Vault access policy. This is a one-line change with significant blast-radius reduction.
6. **[High]** Add `actions/cache` for NuGet in `build` and `tests` jobs. This reduces pipeline duration on every run and is a ~10-line change.
7. **[Medium]** Upgrade `azurerm` provider from `=4.1.0` to `~> 4.1` (or latest `4.x`). Review the azurerm changelog for any resource schema changes affecting `azurerm_container_app` or `azurerm_postgresql_flexible_server`.
8. **[Medium]** Add `lifecycle { prevent_destroy = true }` to Key Vault, PostgreSQL server, and Container App.
9. **[Medium]** Migrate Key Vault authorization from access policies to Azure RBAC (`enable_rbac_authorization = true`). Plan this with care as it requires replacing the access policy resources.
10. **[Medium]** Define at least two metric alerts: PostgreSQL CPU saturation and Container App restart count.
11. **[Medium]** Tighten the Terraform `required_version` constraint to `>= 1.5.2, < 2.0.0`.
12. **[Medium]** Address bootstrap state: store it in Azure Blob Storage using a separate key to eliminate the single-developer local state risk.
